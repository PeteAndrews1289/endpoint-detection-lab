# Incident Response Scenarios

These scenarios describe how a SOC analyst could triage the endpoint alerts generated by this lab.

## Scenario 1: Suspicious PowerShell Alert

### Trigger

A Splunk search identifies PowerShell execution with encoded commands, web request behavior, or unusual parent process context.

### Triage Questions

- Which user launched PowerShell?
- What was the parent process?
- Was the command interactive, scripted, encoded, or remote-content related?
- Did the host make new outbound network connections shortly afterward?
- Did the same user create files, tasks, or accounts after execution?

### Evidence to Review

- Sysmon Event ID 1 process creation
- Sysmon Event ID 3 network connections
- Windows PowerShell logs, if enabled
- File creation or scheduled task events

### Possible Response

- Preserve the command line and parent-child process chain.
- Isolate the host if activity is confirmed malicious.
- Disable or reset the involved account if compromise is suspected.
- Hunt for the same command pattern across other hosts.

## Scenario 2: LSASS Access Alert

### Trigger

Sysmon records a process access event where the target image is `lsass.exe`.

### Triage Questions

- What process accessed LSASS?
- Is the source process expected security software or an unknown binary?
- What user context was used?
- Were there suspicious processes immediately before or after the event?
- Did the host show authentication anomalies afterward?

### Evidence to Review

- Sysmon Event ID 10 process access
- Sysmon Event ID 1 process creation
- Security authentication logs
- Endpoint file path and hash metadata

### Possible Response

- Treat unexpected LSASS access as high priority.
- Collect host timeline and process metadata.
- Rotate exposed credentials if compromise is likely.
- Escalate for deeper endpoint forensic review.

## Scenario 3: New Local Administrator

### Trigger

Windows Security logs show local user creation or local administrator group membership changes.

### Triage Questions

- Who created the account or changed group membership?
- Was the action expected maintenance?
- Did it occur after suspicious shell activity?
- Was the new account used for logon?
- Did the change occur on more than one host?

### Evidence to Review

- Event ID 4720 for account creation
- Event ID 4732 for group membership changes
- Sysmon process creation around the same time
- Logon events for the new account

### Possible Response

- Disable unexpected accounts.
- Remove unauthorized group membership.
- Review recent admin activity on the endpoint.
- Search for the same account name across the environment.

## Scenario 4: Scheduled Task Persistence

### Trigger

Scheduled task creation or `schtasks.exe` execution appears in endpoint logs.

### Triage Questions

- What command does the task execute?
- Which user created the task?
- Is the task name expected?
- Does the task launch PowerShell, cmd, script hosts, or unknown binaries?
- Was the task created after suspicious network or process activity?

### Evidence to Review

- Scheduled task event logs
- Sysmon process creation logs
- Task command/action details
- File metadata for referenced binaries or scripts

### Possible Response

- Disable the task if unauthorized.
- Preserve task XML or command details.
- Search for the same task name or action across other hosts.
- Continue timeline analysis before containment decisions.
